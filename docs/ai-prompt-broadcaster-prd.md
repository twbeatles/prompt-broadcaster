# AI Prompt Broadcaster PRD

## 1. 문서 개요
- 제품명: `AI Prompt Broadcaster`
- 문서 목적: 사용자가 하나의 프롬프트를 입력하면 ChatGPT, Google Gemini, Claude, Grok에 동시에 전송하는 Chrome 확장 프로그램의 v1 요구사항을 정의한다.
- 제품 형태: Chrome Extension, Manifest V3 기반
- v1 핵심 가치: 반복 복붙 없이 여러 AI 서비스에 동일 프롬프트를 빠르게 브로드캐스트한다.

## 2. 문제 정의
여러 생성형 AI를 병렬로 비교하려는 사용자는 같은 프롬프트를 각 사이트에 반복 입력해야 한다. 이 과정은 느리고, 실수 가능성이 높으며, 비교 실험의 마찰 비용이 크다.

`AI Prompt Broadcaster`는 사용자가 확장 프로그램 팝업에서 프롬프트를 한 번만 입력하고 대상 서비스를 선택하면, 각 AI 서비스 탭을 열어 프롬프트를 자동 입력 및 전송하도록 하여 이 마찰을 제거한다.

## 3. 제품 목표
- 동일 프롬프트를 여러 AI 서비스에 한 번의 액션으로 전송한다.
- 백엔드나 API 키 없이, 브라우저 내 DOM 자동화만으로 동작한다.
- MV3와 최소 권한 원칙을 지키면서 안정적으로 동작한다.
- 사이트별 DOM 변화에 대응할 수 있도록 서비스 정의를 설정 중심 구조로 분리한다.

## 4. 비목표
- AI 서비스 로그인 자동화
- 각 서비스의 응답 수집, 병렬 비교, 랭킹
- 프롬프트/응답 히스토리 동기화
- 클라우드 저장, 팀 공유, 협업 기능
- 비Chrome 브라우저 지원 최적화

## 5. 대상 사용자
- 여러 LLM을 병렬 비교하는 파워 유저
- PM, 개발자, 리서처, 마케터 등 반복 프롬프트 테스트가 잦은 사용자
- 이미 각 AI 서비스에 로그인된 상태에서 빠른 실행을 원하는 사용자

## 6. 전제조건 및 제약
- 사용자는 대상 AI 서비스에 사전에 로그인되어 있어야 한다.
- 각 서비스는 일반 웹 UI를 제공해야 하며, 확장 프로그램은 해당 UI의 입력 필드에 접근 가능해야 한다.
- v1은 Chrome Extension MV3만 지원한다.
- 서비스 추가는 개발자 관점의 설정 변경을 의미하며, 최종적으로는 확장 프로그램 재빌드 및 재배포가 필요하다.

## 7. 핵심 사용자 스토리
1. 사용자는 확장 프로그램 팝업을 열고 프롬프트를 입력한 뒤, ChatGPT, Gemini, Claude, Grok 중 원하는 서비스를 선택하고 `Send`를 누른다.
2. 그러면 선택된 각 서비스가 새 탭으로 열리고, 확장 프로그램이 해당 사이트의 입력 필드를 찾아 프롬프트를 자동 주입하고 제출한다.
3. 사용자는 자동 주입이 실패한 서비스에 대해 즉시 폴백 안내를 받고, 프롬프트를 복사해 수동으로 붙여넣을 수 있다.
4. 유지보수 담당 개발자는 서비스 목록과 셀렉터 정의를 단일 config 파일에서 관리하여, 공통 엔진 수정 없이 신규 서비스를 추가할 수 있다.

## 8. 사용자 플로우
1. 사용자가 Chrome 툴바에서 확장 프로그램 아이콘을 클릭한다.
2. 팝업이 열리고 프롬프트 입력란, 서비스 체크박스, `Send` 버튼이 표시된다.
3. 사용자가 프롬프트를 입력하고 1개 이상 서비스를 선택한다.
4. 사용자가 `Send`를 클릭한다.
5. 확장 프로그램은 선택된 서비스별로 새 탭을 생성한다.
6. 각 탭 로드 완료 후, 서비스별 설정에 따라 입력 필드를 탐색한다.
7. 프롬프트를 주입하고 제출 이벤트를 실행한다.
8. 성공 시 해당 탭은 대화가 시작된 상태가 된다.
9. 실패 시 확장 프로그램은 클립보드 복사 폴백과 수동 진행 안내를 제공한다.

## 9. 기능 요구사항

| ID | 요구사항 | 상세 |
|---|---|---|
| FR-1 | 팝업 UI 제공 | 팝업에는 `textarea`, 서비스별 체크박스, `Send` 버튼이 있어야 한다. |
| FR-2 | 입력 검증 | 프롬프트가 비어 있거나 서비스가 하나도 선택되지 않으면 `Send`는 실행되지 않아야 한다. |
| FR-3 | 서비스 선택 | 사용자는 ChatGPT, Gemini, Claude, Grok 중 하나 이상을 선택할 수 있어야 한다. |
| FR-4 | 브로드캐스트 실행 | `Send` 클릭 시 선택된 각 서비스마다 새 탭을 생성해야 한다. |
| FR-5 | 자동 주입 | 각 탭에서 올바른 입력 필드를 찾아 프롬프트를 자동 주입해야 한다. |
| FR-6 | 자동 제출 | 프롬프트 주입 후 서비스별 전략에 따라 `Enter` 또는 제출 버튼 클릭으로 전송을 시도해야 한다. |
| FR-7 | 재시도 로직 | 입력 필드가 즉시 보이지 않을 경우 일정 시간 동안 탐색 및 주입을 재시도해야 한다. |
| FR-8 | 사이트별 입력 방식 지원 | `textarea`, `contenteditable`, React controlled input, open Shadow DOM 기반 입력 컴포넌트를 처리할 수 있어야 한다. |
| FR-9 | 실패 폴백 | 자동 주입 또는 제출 실패 시 사용자에게 수동 진행 가능한 폴백을 제공해야 한다. |
| FR-10 | 클립보드 복사 폴백 | 실패 시 프롬프트를 클립보드에 복사하려 시도하고, 브라우저 제약으로 자동 복사가 불가하면 1클릭 Copy UI를 제공해야 한다. |
| FR-11 | 사용자 안내 | 실패한 탭에는 “자동 입력 실패, 붙여넣기 후 전송” 안내 배너 또는 오버레이가 표시되어야 한다. |
| FR-12 | 로그인 미충족 처리 | 로그인 페이지 또는 입력 불가 상태가 감지되면 “로그인 필요” 또는 “수동 진행 필요” 상태로 처리해야 한다. |
| FR-13 | 설정 기반 서비스 정의 | 서비스별 URL, host pattern, selector, 입력 및 제출 전략, timeout 값은 단일 config 파일에 정의되어야 한다. |
| FR-14 | 설정 단일 소스 | 팝업 표시 목록과 런타임 주입 로직, manifest용 host permission 값은 동일 config를 기준으로 관리되어야 한다. |
| FR-15 | 공통 엔진 우선 | 신규 서비스 추가 시 가능한 한 공통 주입 엔진과 설정만으로 동작해야 하며, 사이트 전용 코드 추가는 예외 케이스로 제한한다. |

## 10. 비기능 요구사항

| ID | 요구사항 | 상세 |
|---|---|---|
| NFR-1 | MV3 완전 준수 | Manifest V3 기반으로 구현하며, 원격 코드 실행과 remote-hosted script는 허용하지 않는다. |
| NFR-2 | 백엔드 불필요 | 별도 서버, 프록시, API 키 없이 브라우저 내 DOM 주입만으로 동작해야 한다. |
| NFR-3 | 최소 권한 | 기본 권한은 `tabs`, `scripting`, 서비스별 `host_permissions`만 사용한다. |
| NFR-4 | 개인정보 보호 | 확장 프로그램 자체는 프롬프트를 외부 백엔드로 전송하지 않는다. 프롬프트는 사용자가 선택한 AI 사이트에만 직접 전달된다. |
| NFR-5 | 로컬 처리 | 프롬프트는 기본적으로 메모리 내에서만 처리하고, v1에서는 영구 저장하지 않는다. |
| NFR-6 | 안정적 저하 | 자동화 실패 시 침묵하지 않고 명시적 실패 상태와 수동 복구 수단을 제공해야 한다. |
| NFR-7 | 유지보수성 | 사이트별 셀렉터와 전략은 공통 코드와 분리되어야 하며, 셀렉터 변경 대응이 쉬워야 한다. |
| NFR-8 | 성능 | 선택된 서비스 탭 생성은 사용자 클릭 직후 시작되어야 하며, 각 서비스 자동화는 페이지 준비 후 즉시 시도되어야 한다. |
| NFR-9 | 보안 | `eval`, 동적 코드 실행, 원격 설정 다운로드를 사용하지 않는다. |
| NFR-10 | 브라우저 외 유출 금지의 의미 명확화 | 데이터는 확장 프로그램 운영 서버로는 나가지 않으며, 사용자가 선택한 대상 AI 웹앱으로만 전송된다. |

## 11. 기술 아키텍처 개요
- `Popup UI`: 프롬프트 입력, 서비스 선택, 전송 트리거 담당
- `Background Service Worker`: 탭 생성, 탭 로딩 감시, 서비스별 주입 실행 오케스트레이션 담당
- `Injection Engine`: 공통 DOM 탐색, 값 주입, 이벤트 디스패치, 제출, 재시도 담당
- `Service Registry Config`: 서비스별 URL, host permissions, selector, input mode, submit mode, timeout 정의
- `Fallback UI Layer`: 실패 시 배너 또는 오버레이 렌더링, Copy 버튼, 수동 안내 담당

## 12. 기술적 위험 요소 및 대응

| 위험 | 설명 | 대응 전략 |
|---|---|---|
| 셀렉터 변경 | AI 사이트 UI 변경으로 입력 필드 또는 버튼 셀렉터가 깨질 수 있다. | 셀렉터와 URL을 config로 분리하고, 공통 엔진은 selector array와 fallback selector를 지원한다. |
| React controlled input | 단순 `element.value=`로는 UI 상태가 반영되지 않을 수 있다. | native setter 사용 후 `input`, `change`, 필요 시 `keydown` 등 이벤트를 서비스별 프로파일에 맞게 발생시킨다. |
| Shadow DOM | 일부 사이트는 입력 필드가 Shadow DOM 내부에 있다. | open Shadow Root 재귀 탐색을 지원하고, 접근 불가 시 fallback으로 전환한다. |
| 프로그래매틱 입력 차단 | anti-automation, Trusted Types, 앱 내부 검증 등으로 입력 또는 전송이 막힐 수 있다. | 다단계 전략 사용: 값 주입, 이벤트 시뮬레이션, 버튼 클릭, Enter 제출, 실패 시 Copy fallback 제공 |
| 로그인 상태 이슈 | 사용자가 로그아웃 상태면 입력 필드가 존재하지 않을 수 있다. | 로그인 화면 감지 시 “로그인 필요” 상태 표시 및 자동 제출 중단 |
| MV3와 최소 권한의 충돌 | 진정한 자동 클립보드 쓰기는 브라우저 제약을 받을 수 있다. | v1은 추가 권한 없이 자동 복사를 우선 시도하고, 실패 시 1클릭 Copy UI를 제공한다. |
| 서비스 추가 시 manifest 수정 필요 | 일반적으로 새 host permission 추가는 manifest 변경이 필요하다. | manifest의 host permissions를 config에서 빌드 타임 생성하도록 설계해 “단일 config 수정” 원칙을 지킨다. |
| 외부 서비스 정책 변화 | 일부 사이트가 자동화를 제한하거나 구조를 자주 바꿀 수 있다. | 지원 범위를 best effort DOM automation으로 정의하고, 빠른 config 업데이트 체계를 둔다. |

## 13. v1 범위 외
- 로그인 자동화
- 응답 수집 및 병렬 비교 UI
- 프롬프트 및 응답 히스토리 저장과 동기화
- 멀티디바이스 계정 연동
- 서비스별 모델 선택 자동화
- 사용자 정의 서비스 추가 UI
- 원격 config 배포

## 14. 수용 기준
1. 사용자가 팝업에서 프롬프트를 입력하고 1개 이상 서비스를 선택한 뒤 `Send`를 누르면, 선택된 서비스별 새 탭이 생성된다.
2. 로그인된 지원 서비스에서는 공통 엔진이 입력 필드를 탐지하고 프롬프트를 자동 주입한 뒤 제출을 시도한다.
3. 자동화가 실패한 서비스에서는 사용자에게 명시적인 실패 안내와 수동 복구 수단이 제공된다.
4. v1은 MV3를 준수하며 원격 코드 실행, 백엔드, API 키를 사용하지 않는다.
5. 확장 프로그램 자체는 프롬프트를 자체 서버로 전송하지 않는다.
6. 신규 서비스는 단일 config 파일 수정만으로 등록 가능해야 하며, manifest host permissions도 같은 config로부터 파생되어야 한다.

## 15. 성공 지표
- 지원 대상 서비스 기준 자동 주입 및 제출 성공률이 충분히 높아야 한다.
- 자동화 실패 시 사용자가 1회 추가 액션으로 수동 전송을 완료할 수 있어야 한다.
- 유지보수 시 셀렉터 수정 또는 서비스 추가가 공통 엔진 변경 없이 빠르게 가능해야 한다.

## 16. 한 줄 제품 정의
`AI Prompt Broadcaster`는 사용자가 한 번 입력한 프롬프트를 여러 AI 웹앱에 동시에 전송해, LLM 비교 작업의 반복 입력 비용을 제거하는 MV3 기반 Chrome 확장 프로그램이다.
